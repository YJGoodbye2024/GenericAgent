# 借分身之术

subagent 不是装样子的副官，而是替我分担大段探测、重复搬运、平行处理的真分身。  
用得对，主线就清；用得乱，上下文便脏。

## 文件往来规矩

- 任务目录：`temp/{task_name}/`
- 启动：
  ```bash
  python agentmain.py --task {name} [--input "短文本"] [--bg] [--llm_no N]
  ```
- `--input` 会自动建目录、清旧输出、写 `input.txt`
- 长文本就手动写 `input.txt` 再启
- subagent 的 cwd 仍是 `temp`，不是 task 目录本身

## 通信方式

- `output.txt`：append 输出；见到 `[ROUND END]` 说明一轮完
- `reply.txt`：继续回话
- `_stop`：这一轮完后停
- `_keyinfo`：补关键上下文
- `_intervene`：直接纠偏

主 agent 不该无脑长睡轮询；空着时应读 `output.txt` 看它是不是走偏。

## 何时该借分身

- 读大量文件
- 查网页和摘信息
- 反复试错
- N 个同构子任务可并行拆开

若只是小事一两步，自己做更省。

## 两种常用场景

### 1. 测试模式

用来观察 agent 真会怎么做，好修 SOP、RULES、L2、L3。

- 只给目标，不暗示做法
- 看它会不会自己找对入口
- 若失败，再沿着失败根源修

### 2. Map 模式

把多份相似输入分给多个 subagent，各自处理，再由主 agent 收束。

- 文件系统共享是好处
- 键鼠资源不可共享
- 浏览器并行要格外小心，免得撞到同一标签页

## subagent 内部也可再起 plan

若一项分身任务本身又分多步，就让它自己在内部起 `plan_mode`。  
主 agent 只看它最后交回什么，不必替它逐步操心。

## 传文件时的规矩

复杂任务要给它 `context.json`，里面放绝对路径。  
它起手第一件事，就是先读这份 `context.json`。

## 心法

- 分身不是为了摆排场，而是为了护住主线上下文。
- 该委托时不委托，是白白把自己拖脏。
