# 谋划之术：Plan Mode

这门术是为长任务、分枝路、并行事而设。  
一两步能了的事，不必铺纸张罗；三步以上且互有关联的事，就该先立计划，再动手。

## 何时起这门术

- 三步以上，有依赖
- 多文件协同
- 有条件分支
- 适合并行委托

## 起手

1. 先建工作目录：`./plan_XXX/`
2. 再用：
   ```python
   handler.enter_plan_mode("./plan_XXX/plan.md")
   ```
3. 更新 checkpoint，记下任务、约束、已匹配的 SOP、当前处于探索态

## 第一段：探索态

在真正写计划前，先弄清三件事：

- 眼前环境如何
- 有哪些可用旧术
- 关键不确定点是什么

### 规则

- 主 agent 不直接做大段环境探测
- 探测优先委托 subagent
- subagent 只读，不改，不做有副作用的动作
- 若探索 subagent 起不来，先排查再重试，最多两回

### 探索产物

必须落一份 `exploration_findings.md`，至少写：

- 环境现状
- 关键发现
- 风险 / 未明处

## 第二段：写 `plan.md`

`plan.md` 至少要有：

- 任务标题
- 一句话需求与约束
- 探索发现
- 分步执行计划
- 验证检查点

### 标记规则

- `[ ]`：待做
- `[✓]`：已做并写了结果
- `[D]`：应委托 subagent 的搬砖活
- `[P]`：并行 map 模式
- `[?]`：条件分支

### 何时标 `[D]`

- 需读大量文件或代码
- 需浏览网页大量取材
- 需重复试错
- 需跑测试并解释结果

## 第三段：执行循环

每一轮都按同一顺序来：

1. `file_read(plan.md)` 找第一个 `[ ]`
2. 该步若有 SOP，先读 SOP
3. 若是 `[D]` 或 `[P]`，按 `subagent.md` 办
4. 执行
5. 做最小验证
6. 用 `file_patch` 把 `[ ]` 改成 `[✓ 简要结果]`
7. 立刻回到第 1 步

### 执行态禁令

- 禁凭记忆做下一步
- 禁跳过验证
- 禁没做终止检查就声称完成
- 禁做完一步后停下来写大段空汇报

## 第四段：验证态

凡完整的 plan 任务，都要再起一个独立验证 subagent。

### 验证前准备

在 `plan_XXX/` 下建 `verify_context.json`，至少含：

- task_description
- plan_file
- task_type
- deliverables
- required_checks

### 验证 subagent 要做的事

- 先读 `memory/skills/verify_sop.md`
- 站在“证明你交付物不可靠”的位置去对抗性验证
- 输出 `VERDICT: PASS / FAIL / PARTIAL`

## 最后的终止检查

必须重新通读 `plan.md`，确认没有残留 `[ ]`。  
真到这一步，才算把这门术行完。
