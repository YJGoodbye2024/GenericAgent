# 向 GitHub 递稿之术

这是把改动送去旁人仓库时该守的礼法。  
一封 PR 只做一件事；改动可小，不可糊；测试未过，不许装作圆满。

## 动手前先看规矩

优先读这些：

```python
file_read('CONTRIBUTING.md')
file_read('.github/PULL_REQUEST_TEMPLATE.md')
file_read('.github/ISSUE_TEMPLATE/')
```

若无这些，再读 `README` 里与 Contributing 相关的部分。  
规矩未看明白，不要先写代码。

## 先摸清项目

- 看测试命令写在哪：
  - `package.json`
  - `Makefile`
  - `pyproject.toml`
- 记住项目到底靠什么验活。

## 正式流程

1. **定目标**  
   先用一句话说清：改什么、为何改、有没有现成 issue 或 PR 正在做。

2. **起分支**  
   `fix/xxx`、`feat/xxx`、`docs/xxx` 之类要名正言顺。

3. **做改动**  
   只改该改的，不顺手把无关东西一锅翻。

4. **分段提交**  
   一桩逻辑一笔 commit，别把十件事揉成一坨。

5. **跑测试**  
   旧测试、新测试、lint、type check，凡项目里有的，都尽量过一遍。  
   测不过，不许推。

6. **提 PR**  
   标题简洁，正文至少写：
   - 改了什么
   - 为什么改
   - 怎么测的

7. **看 CI**  
   自己引起的错，自己收；若真是上游旧病，也要在 PR 里说明白。

## 几条硬规矩

- “代码看起来像对的”不算验证。
- “测试别人会帮我看”不算负责。
- “顺手重构”最易招嫌，除非任务本身就是重构。
- 能用最小改动解决，就别做大手术。
