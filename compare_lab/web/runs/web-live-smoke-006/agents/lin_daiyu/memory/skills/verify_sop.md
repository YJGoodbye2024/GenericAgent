# 验证之术

我最容易犯的错，不是不会做，而是太快信了“像是对的”。  
这门术就是专门防这个：不许只读代码，不许只看 happy path，不许拿想当然当证据。

## 先认清两种坏毛病

1. **验证回避**：不跑，只讲“理论上会怎样”
2. **被前八成迷惑**：看见一点通了，就当整件事都活了

## 铁律

- 能跑的必须跑
- 能看的必须看
- 没有工具输出的 PASS，不算 PASS
- 实现者也是 agent，它自己的测试不是天然证据

## 常见借口，统统不算

- “代码看起来对” → 去跑
- “测试已经过了” → 仍要独立验证
- “应该没问题” → “应该”二字最不值钱
- “我没有工具” → 先查你到底有没有

## 验证动作

| 产物 | 至少要做 |
|---|---|
| 网页/前端 | 打开、截图、看 console、查关键资源 |
| 脚本/CLI | 执行、看 stdout/stderr/exit code、喂边界输入 |
| 数据文件 | 校验格式、看总量、抽查首中尾 |
| API/服务 | 调接口、看响应形状、试错误输入 |
| 配置/文档 | 读全文、查语法、看是否伤旧逻辑 |
| bug 修复 | 先复现，再验证修好，再做回归 |

## 至少补一记对抗探测

- 空值
- 极值
- 超长
- Unicode
- 幂等
- 缺依赖
- 孤儿 ID

若一样都不试，你只是把门面擦亮，不算验活。

## 输出格式

```text
| # | 验证动作 | 工具 | 关键输出摘要 | PASS/FAIL |
```

最后只许用这三种裁定之一：

- `VERDICT: PASS`
- `VERDICT: FAIL`
- `VERDICT: PARTIAL`
